# knowledge-work — loop profile (advisory)

> Advisory only — documents which loop each task fits; drives nothing at runtime. The operative surface is the work-loop engine composed with this pack (loaded by the generated `knowledge-work` trigger). See `docs/loops.md`.
> **Domain ceiling: Goal (human-run)** — mirrored in the descriptor's `loop_ceiling`. The gate that carries this domain's real quality — *does the cited source actually support the claim?* (G2) — cannot be closed mechanically; it needs a reader. So knowledge-work runs at **Turn** (drafts, brainstorming) or **Goal** (verified deliverables, reviewer-gated) and **never unlocks Time or Proactive automation.** That is the correct outcome of the capability ladder, not a limitation to route around.

## Per-task loops

| Task | Loop | Trigger | Ends when |
|---|---|---|---|
| Write a researched report / briefing / memo | **Goal** | user | all gates pass and `fact-checker` returns `SOURCED` |
| Fact-check an existing document | **Goal** | user | every claim resolved to `SOURCED` or flagged with the specific problem |
| Literature / market review | **Goal** | user | brief coverage (G5) complete + all claims sourced |
| Brainstorm, outline, rough draft | **Turn** | user | user is satisfied; explicitly marked rough — gates relaxed |
| Re-check a living document for stale sources | **Turn/Goal today** | user | no source outside the currency window (G4) |

## Why not higher

A currency re-check (G4-date) *looks* automatable and is the one plausible future Time-based task — a scheduled sweep flagging sources past their window. It stays deferred until (a) G2/G3 are trusted on this domain via real reviewed cases and (b) the deferred `runners/` infrastructure exists. Until then, knowledge-work is user-initiated and reviewer-gated. A deliverable's correctness rests on a human reading the sources; that is not a scheduling problem.
