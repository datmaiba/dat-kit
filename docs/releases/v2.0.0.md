# dat-kit v2.0.0 — the open platform

v2.0.0 rebuilds dat-kit as a registry-driven, multi-domain, multi-host
platform. It replaces the single-domain build-loop skeleton with a
three-layer architecture: a domain-agnostic work-loop **engine**, six-slot
**Domain Packs** that supply the actual working discipline for a field, and a
lifecycle-governed **Host Adapter** layer that lets any AI coding host consume
the same canonical contract without duplicating policy.

Nothing in this release changes what dat-kit asks a human to approve. It
changes how much of dat-kit's own behavior is now generated from a single
source of truth instead of hand-maintained per host and per domain.

## What changed

**Three-layer architecture.** The work-loop engine (`engine/work-loop/`) is
extracted from `build-loop` and owns the domain-agnostic phase sequence
(context load → self-question → plan → approval gate → build → verify →
review → harvest). Domain Packs compose on top of it.

**Six-slot Domain Packs.** `software-dev` (the original build-loop discipline)
and `knowledge-work` (research, writing, fact-checked deliverables) are now
both active six-slot packs: workflow, ground truth, gates, reviewers,
deliverables, and loop profile. Their registry descriptors bind the packs to
the engine, and their thin trigger skills are generated projections. A
synthetic non-software pack (bookkeeping/ledger-close, fixture-only) proved
the engine composes for domains dat-kit does not yet ship.

**Registry-driven adapters.** Host support (`claude-code`, `codex`,
`cursor`, `gemini-cli`) is now declared in a registry with a typed four-state
lifecycle (`repo_only → migration_ready → scaffold_active → retired`) and
dated official facts, rather than hand-written per-host prose. A synthetic
Host Adapter proved a new host needs no validator or shell edit to onboard.

**`AGENTS.md` as sole policy owner.** Every adapter declares
`policy_prohibition.canonical_owner: "AGENTS.md"` and ships only as a thin,
lifecycle-governed pointer (`.claude/CLAUDE.md`, `CLAUDE.md`, `.cursorrules`
→ `.cursor/rules/dat-kit.mdc`). This was true in 1.16 for Claude Code; it is
now true and mechanically enforced (`test_adapter_conformance.py`) across all
four hosts.

**Deterministic, fail-closed registry bootstrap.** Malformed, old, unknown,
future, or mixed-format registry data is rejected with a named diagnostic
before any file is generated — never a partial or silently-degraded scaffold.

**Byte-exact, checked projections.** The two projection kinds in v2.0 — thin
domain-trigger skills and the scaffold manifest — are checked byte-for-byte
against what `render.py` would produce; `render.py --check` fails closed on
drift instead of silently regenerating. Adapter pointers remain
lifecycle-governed static templates consumed through the manifest.

## Migrating from v1.16

`dat-kit 1.16.0` is recognized only as a migration source — never a green
revision. Existing 1.16 projects fail closed with `CONTRACT_MIGRATION_REQUIRED`
and a deterministic, read-only migration plan; project-owned policy
(`AGENTS.md` customizations, `docs/agent-working-rules.md`) is preserved by
semantic merge, never overwritten.

**Full walkthrough, including the exact failure diagnostics
(`CONTRACT_MIGRATION_REQUIRED`, `COMPETING_AGENTS`,
`BROWNFIELD_CONTRACT_CONFLICT`) and the backup/rollback path:
[`docs/releases/migration-2.0.md`](migration-2.0.md).**

The migration path was proven against a clean v1.16 fixture, a customized
v1.16 fixture (local template edits + user spec content), and one real v1.16
project (an isolated clone of a production repo) — all three migrated cleanly
to exit 0. Project-owned policy was preserved in every case: verified present
after the semantic merge in the customized fixture, and verified
byte-identical before/after (sha256) in the real-project run. A live rollback
rehearsal to v1.17.1 tooling confirmed the round trip never corrupts either
era's project files.

## Known limitations

This release ships with two limitations named openly rather than silently
dropped or discovered later:

1. **The Cursor host adapter has no live-evidence checklist gathered for this
   release.** `cursor` remains at lifecycle `migration_ready`, with its
   declared artifacts and typed `RETIRE_LEGACY` migration semantics
   mechanically tested (`test_adapter_conformance.py`, migration-planner
   tests). `adapters/cursor/ADAPTER.md` documents that no scriptable headless
   Cursor exists, so live verification is a manual checklist rather than an
   automated gate, and it was not run for v2.0.0 because Cursor was not
   available on the machine that produced this release. Nothing about Cursor
   support is unverified at the *mechanical* level — only the manual,
   human-in-the-editor checklist is deferred. Track and close this in a
   v2.0.x follow-up.
2. **A registry-internal inconsistency between two adapters' `repo_only`
   declarations.** `codex` is `repo_only` with `project_artifacts: []`
   (matching the documented definition: "no project artifact exists or is
   emitted"), while `gemini-cli` is also `repo_only` but declares a
   `GEMINI.md` project artifact — contradicting that same definition. This
   has no runtime effect (`gemini-cli` scaffolds nothing regardless), and it
   is registry data, not generated-project behavior. The fix requires editing
   frozen registry data and follows the format-freeze amendment procedure
   after this tag, not before it.

## Compatibility and rollback

`release/1.x` continues to receive its own maintenance line independent of
this release; its own gates were re-verified on its own branch as part of
this release's exit criteria. Rolling back to v1.17.1 tooling is safe and
proven live: v1.17.1 tooling refuses to touch a v2.0-scaffolded project
(fails closed with `COMPETING_AGENTS` / `BROWNFIELD_CONTRACT_CONFLICT`,
every byte preserved), and a v1.16 project that passes through a rollback
round trip remains a clean forward-migration source afterward.
`docs/codex.md` is kept as a redirect stub specifically because v1.17.1's own
error text still points there — do not remove it while v1.17.1 remains a
supported rollback target.
